# Tuist Project Structure

This document describes the Tuist setup used in this repository for Swift exercise projects. It was defined through a deliberate design session and should be the reference for any AI or developer continuing this work.

---

## Context

This repository contains programming exercises from competitive coding platforms (URI/Beecrowd, LeetCode, HackerRank, etc.), solved in multiple languages. For Swift specifically, the author wants:

- A **CommandLine** target to run exercises manually
- A **unit test** target to validate exercises automatically
- A **Tuist-managed** Xcode project (no manual `.xcodeproj` maintenance)
- A **template** that can be reused to bootstrap new sites with a single command

## Tuist Targets

Two targets are defined. There is **no Framework target** — source sharing is achieved by including `Application/Sources/` in both targets directly.

### 1. CommandLine Target (`siteName`)

- Product: `.commandLineTool`
- Sources: `Application/Sources/**`
- Entry point: `main.swift`
- Used to run exercises manually from Xcode or terminal

### 2. Test Target (`siteNameTests`)

- Product: `.unitTests`
- Sources:
  - `Application/Sources/**` **excluding** `Application/Sources/main.swift`
  - `Application/Tests/**`
- No dependency on a framework — sources are compiled directly into the test bundle
- `main.swift` is excluded to avoid entry point conflict with the test runner

## Tuist.swift

```swift
import ProjectDescription

let tuist = Tuist(
    project: .tuist(
        compatibleXcodeVersions: .upToNextMajor("26"),
        swiftVersion: "5.10"
    )
)
```

## Project.swift

```swift
import ProjectDescription

// MARK: - Constants

let siteName = "URI"          // Change this per site: "LeetCode", "HackerRank", etc.
let bundlePrefix = "com.gui"
let deploymentTarget = "14.2"

// MARK: - Helpers

func commandLineTarget() -> Target {
    .target(
        name: siteName,
        destinations: .macOS,
        product: .commandLineTool,
        bundleId: "\(bundlePrefix).\(siteName.lowercased())",
        deploymentTargets: .macOS(deploymentTarget),
        sources: ["Application/Sources/**"]
    )
}

func testTarget() -> Target {
    .target(
        name: "\(siteName)Tests",
        destinations: .macOS,
        product: .unitTests,
        bundleId: "\(bundlePrefix).\(siteName.lowercased()).tests",
        deploymentTargets: .macOS(deploymentTarget),
        sources: [
            .glob("Application/Sources/**", excluding: ["Application/Sources/main.swift"]),
            "Application/Tests/**"
        ]
    )
}

// MARK: - Project

let project = Project(
    name: siteName,
    organizationName: "Gui",
    targets: [
        commandLineTarget(),
        testTarget()
    ]
)
```

## Template System

### Template.zip

Located at `Templates/Swift-Project/Template.zip`. Contains the base project structure with placeholder name `"NewProject"` (used internally by the zip structure). All `let siteName` values in the Swift files use `"Template"` as the placeholder that gets replaced by the script.

### new-project.sh

```
Usage: ./new-project.sh <SiteName>
```

What it does:
1. Checks if `Exercises/<SiteName>` exists — creates it if not
2. Errors if `Exercises/<SiteName>/Swift` already exists
3. Unzips `Template.zip` into `Exercises/<SiteName>/`
4. Renames the extracted `NewProject/` folder to `Swift/`
5. Runs `sed` to replace all occurrences of `"Template"` with `<SiteName>` in all `.swift` files

After running the script, go to the new folder and run:
```bash
tuist generate --no-open
```

## Why No Framework?

The original URI project used a Framework target so that the test target could do `@testable import UriFramework`. This was the only way to share code between the CommandLine and test targets in plain Xcode.

With Tuist, both targets can reference the same source files directly via `sources:` globs. This eliminates the need for a Framework entirely, reducing the project from 3 targets to 2.

The test files **do not need any import** to access exercise functions — they are compiled directly into the test bundle.

## Running Tests from CLI

```bash
cd Exercises/<SiteName>/Swift
xcodebuild -workspace <SiteName>.xcworkspace \
           -scheme <SiteName> \
           -destination 'platform=macOS' \
           -only-testing:<SiteName>Tests \
           test
```

## InputProvider: Key Design Decision

The `InputProvider.swift` in `Aux/Providers/` is the core abstraction that makes exercises testable. **Do not replace it with a simpler version** — the real implementation has specific behavior that tests depend on.

See `exercise-pattern.md` for full details on how it works and how to use it.
